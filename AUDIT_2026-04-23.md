# XRPL Arbitrage Bot Audit (2026-04-23)

## Scope

Audited latest committed code (`7de41cd`) with focus on "no trades firing" behavior.

## High-confidence findings

1. **Critical execution cap in atomic leg-1 amount sizing**
   - In `src/executor.py`, leg 1 (`XRP -> IOU`) uses partial payment semantics where `Amount` is a **ceiling**.
   - The previous code set `Amount.value = opportunity.input_xrp`, which is not a safe ceiling for many markets.
   - Example: if an asset costs `0.4 XRP/IOU`, then 1 XRP can buy ~2.5 IOU; ceiling of `1.0 IOU` clips execution and starves leg 2.
   - Symptom alignment: repeated `leg2_sim` rejections (often `tecPATH_PARTIAL`) even with low thresholds.
   - Fix applied: `LEG1_AMOUNT_BUFFER_MULTIPLIER=1000` and `Amount.value=input_xrp*1000` to avoid clipping while retaining tfPartialPayment control.

2. **Threshold tuning confusion can look like "bot broken"**
   - The scanner now uses `get_profit_threshold(currency)` (tiered high/low liquidity thresholds) instead of a single `PROFIT_THRESHOLD` everywhere.
   - Lowering only `PROFIT_THRESHOLD` may have little/no effect for many currencies when `PROFIT_THRESHOLD_LOW_LIQ` remains high.
   - This can make operators think thresholds were lowered but opportunities still never pass.

3. **Test harness not runnable as-is in this environment**
   - Async tests are marked with `@pytest.mark.asyncio` but `pytest-asyncio` is not installed in the current environment, causing broad false failures.
   - This blocks local validation confidence unless dependency parity is fixed.

## Additional observations (non-blocking but relevant)

- `README.md` still emphasizes a single threshold model in some sections, while runtime logic is tiered.
- `scripts/diagnose_two_leg.py` prints `PROFIT_THRESHOLD` in summary, which may not reflect actual per-currency gating logic.
- Full scan runs every `SCAN_INTERVAL` ledgers and event-driven scans depend on `book_changes`; if stream provider behavior degrades, opportunity cadence drops sharply.

## Recommended next operator checks

1. Tail logs for per-opportunity rejection reason counts (`leg1_sim`, `leg2_sim`, `single_writer_violation`, etc.).
2. Run `scripts/diagnose_two_leg.py` against the same wallet/node and compare gross spreads vs net threshold by currency class.
3. Explicitly set all three threshold vars during tuning:
   - `PROFIT_THRESHOLD_HIGH_LIQ`
   - `PROFIT_THRESHOLD_LOW_LIQ`
   - `PROFIT_THRESHOLD` (for scripts/backward references)
4. Keep `XRPL_WS_URL` / `XRPL_RPC_URL` on full-history full-rippled endpoints (not Clio-only for path-dependent ops).
